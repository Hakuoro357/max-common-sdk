# Спецификация support matrix для SDK MAX

## 1. Цель

Определить единую матрицу поддержки SDK, чтобы было прозрачно:

- какие языки поддерживаются официально
- какие пакеты доступны для каждого языка
- какой у них статус зрелости
- какой API coverage уже закрыт
- какие языки находятся в приоритете

## 2. Зачем нужна support matrix

При росте числа SDK по языкам быстро появляется расхождение между:

- реальным состоянием пакетов
- документацией
- ожиданиями интеграторов

Support matrix нужна как единая точка правды для:

- product planning
- developer relations
- release management
- roadmap приоритизации

## 3. Что должна показывать матрица

Для каждого языка и пакета должны фиксироваться:

- язык
- тип пакета
- статус
- ownership
- supported API version
- feature coverage
- способ публикации
- дата последнего релиза
- целевой приоритет

## 4. Категории пакетов

Для каждого языка матрица должна минимум различать:

- `client`
- `bot-sdk`
- `webhook`

При необходимости могут быть дополнительные типы:

- `miniapp-ui`
- `testing`
- `framework-adapter`

## 5. Статусы зрелости

Рекомендуемые статусы:

- `planned`
- `experimental`
- `beta`
- `stable`
- `deprecated`

### Правила статусов

`planned`
- пакет еще не выпущен
- может быть RFC, дизайн или roadmap

`experimental`
- пакет доступен для раннего использования
- возможны breaking changes без длинного переходного периода

`beta`
- пакет пригоден для пилота и ограниченного production
- API в основном стабилен

`stable`
- пакет рекомендован для production
- breaking changes проходят через formal deprecation process

`deprecated`
- пакет не рекомендуется для новых интеграций
- должен быть указан migration path

## 6. Ownership model

У каждой записи в support matrix должен быть владелец:

- core platform team
- language owner
- external maintainer

Если пакет не имеет явного owner, он не должен маркироваться как `stable`.

## 7. Feature coverage

Поддержка должна оцениваться не только по наличию репозитория, но и по coverage.

Минимальные категории coverage:

- auth
- basic messages
- updates/webhooks
- file upload
- file download
- pagination
- error mapping
- retry hooks
- middleware hooks
- testing helpers

Рекомендуемые уровни:

- `none`
- `partial`
- `full`

## 8. Рекомендуемая стартовая матрица

| Язык | Client | Bot SDK | Статус | Приоритет | Комментарий |
|---|---|---|---|---|---|
| TypeScript | Да | Да | stable | high | Базовый web/backend сценарий |
| Go | Да | Частично | beta | high | Сильный backend/use-case для ботов |
| Java | Да | Да | beta | medium | Нужна модернизация и выравнивание DX |
| Python | Да | Частично | experimental | high | Нужен официальный first-party SDK |
| C# | Нет | Нет | planned | highest | Главный пробел для enterprise |
| PHP | Нет | Нет | planned | high | Важен для web-интеграций и legacy |
| Kotlin | Нет | Нет | planned | high | Нужен для modern JVM и Android-friendly сегмента |
| Rust | Нет | Нет | planned | medium | Полезен для infra/high-load, но не первый |

## 9. Приоритет запуска языков

Рекомендуемый порядок развития официальной поддержки:

1. `Python` довести до официального stable-пакета
2. `C#`
3. `PHP`
4. `Kotlin`
5. `Rust`

Обоснование:

- `Python` уже востребован и частично присутствует в экосистеме
- `C#` закрывает сильный enterprise-сегмент
- `PHP` дает широкий охват интеграций
- `Kotlin` усиливает современный JVM-сегмент
- `Rust` важен, но менее массовый

## 10. Источник данных для матрицы

Support matrix не должна вестись вручную только в документации.

Желательно, чтобы она собиралась из структурированных данных:

- package metadata
- release metadata
- CI status
- docs annotations
- ownership registry

Например, через единый YAML/JSON-реестр, из которого генерируется документация.

## 11. Требования к обновлению

Матрица должна обновляться:

- при создании нового SDK
- при смене статуса зрелости
- при выходе major/minor release
- при смене owner
- при деприкации пакета

Если релиз опубликован, а support matrix не обновлена, процесс считается незавершенным.

## 12. Публичное отображение

Support matrix должна быть видна:

- в документации разработчика
- в репозитории SDK platform
- в release notes при изменении статуса

Пользователь должен быстро понимать:

- какой язык поддерживается официально
- какой статус у SDK
- подходит ли он для production

## 13. Итог

Support matrix для MAX SDK должна стать обязательным артефактом платформы, а не вспомогательной таблицей.

Ключевой принцип:

> Поддержка языка считается существующей только тогда, когда она явно отражена в матрице, имеет owner, статус и понятный coverage.
