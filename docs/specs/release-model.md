# Спецификация release model для multi-language SDK MAX

## 1. Цель

Определить модель релизов, при которой:

- API развивается предсказуемо
- SDK по разным языкам публикуются независимо
- breaking changes контролируются централизованно
- документация и generated code всегда синхронизированы

## 2. Основной принцип

Версии должны существовать на трех уровнях:

- `API version`
- `Generator version`
- `SDK package version`

Эти уровни нельзя жестко склеивать в одну общую цифру.

## 3. Модель версий

### 3.1 API version

Отвечает за публичный контракт MAX Bot API:

- endpoints
- request / response schema
- webhook/update payload
- auth contract
- error model

Рекомендуемая модель:

- `v1`, `v2` для major-уровня
- minor-изменения внутри major без ломки контракта

Breaking change в API допустим только при повышении major-версии API.

### 3.2 Generator version

Отвечает за:

- шаблоны генерации
- структуру generated code
- генерацию документации
- генерацию tests/fixtures

Generator может обновляться независимо от API, если:

- не ломает публичный SDK surface
- не меняет контракт сериализации без явного основания

### 3.3 SDK package version

Каждый пакет версионируется отдельно.

Примеры:

- `@max/client`
- `@max/bot-sdk`
- `Max.BotApi`
- `Max.BotSdk`
- `max-bot-client`

Это позволяет:

- релизить багфиксы отдельно по языкам
- выпускать улучшения DX без задержки остальных SDK
- не блокировать всю платформу одной проблемой в одном языке

## 4. Правила изменения версий

### Major

Повышается, если:

- меняется публичный API пакета с breaking effect
- удаляются или переименовываются публичные методы
- ломается совместимость конфигурации
- меняются ключевые runtime semantics

### Minor

Повышается, если:

- добавлены новые API methods
- добавлены новые расширяемые возможности
- добавлены новые middleware hooks
- добавлена поддержка нового non-breaking поведения

### Patch

Повышается, если:

- исправлены баги
- исправлена генерация
- улучшена документация без смены контракта
- исправлены тесты, примеры, packaging, metadata

## 5. Release train

Рекомендуется 3 типа релизов.

### 5.1 Scheduled release

Плановый релиз по календарю:

- предсказуем для интеграторов
- удобен для накопления minor-изменений

### 5.2 Hotfix release

Внеплановый релиз:

- баги в runtime
- критические проблемы сериализации
- проблемы auth / retries / uploads

### 5.3 API rollout release

Релиз, привязанный к изменению API:

- новый endpoint
- новые поля
- новые event types
- изменения webhook schema

Такой релиз должен проходить через compatibility gate.

## 6. Release gates

Перед публикацией любого SDK должны быть обязательны:

1. schema validation
2. API diff
3. backward compatibility check
4. generated code diff review
5. contract tests
6. language-specific unit/integration tests
7. docs/examples sync check
8. package metadata validation

Без прохождения этих этапов релиз не публикуется.

## 7. Совместимость

### 7.1 Backward compatibility

По умолчанию все изменения считаются backward-compatible, пока не доказано обратное.

Нарушениями считаются:

- удаление поля
- сужение допустимых значений
- смена обязательности поля
- изменение типа поля
- изменение поведения ошибки без отражения в contract

### 7.2 Deprecation policy

Для breaking migration требуется стадия deprecation:

- пометка как deprecated
- указание replacement path
- срок удаления
- ссылка на migration guide

Нельзя удалять публичное поведение без заранее объявленного переходного периода, кроме security-критичных случаев.

## 8. Каналы публикации

Каждый язык публикуется в свой реестр:

- npm
- PyPI
- Maven Central
- NuGet
- Packagist

Публикация должна быть автоматизирована, но gated.

## 9. Документация релизов

Для каждого релиза обязательно:

- release notes
- список breaking/non-breaking изменений
- поддерживаемая версия API
- migration notes
- обновленные examples при необходимости

## 10. Поддержка нескольких языков

Релизы языков не обязаны быть синхронными по дате, но обязаны быть синхронными по contract support.

Это значит:

- новый API endpoint может выйти сначала в одних SDK
- но матрица поддержки должна быть прозрачной
- в docs должно быть видно, в каких языках endpoint уже доступен

## 11. Рекомендуемая модель статусов

Для каждого SDK и пакета нужен явный статус:

- `experimental`
- `beta`
- `stable`
- `deprecated`

Статус должен публиковаться в docs и package metadata.

## 12. Матрица поддержки

Нужно вести явную support matrix:

- язык
- пакет
- статус
- поддерживаемая версия API
- дата последнего релиза

Это важнее, чем пытаться держать все пакеты на одной и той же версии.

## 13. Итог

Release model для MAX SDK должен опираться на независимые версии и жесткие compatibility gates.

Ключевой принцип:

> API единый, релизная дисциплина единая, публикация пакетов независимая.
